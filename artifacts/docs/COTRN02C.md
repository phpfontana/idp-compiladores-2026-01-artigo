# COTRN02C

## Purpose
COTRN02C is the CICS pseudo-conversational transaction add program for CardDemo. The user provides either an account ID or a card number (the program resolves the other via CXACAIX/CCXREF cross-reference), fills in all transaction fields (type, category, source, amount, description, original date, processing date, merchant details), and confirms with `'Y'` to write a new record to TRANSACT. Dates are validated via `CSUTLDTC`. Trans-ID is `CT02`.

## Inputs
- `COTRN2AI` ‚Äî BMS map input from COTRN02 mapset: `ACTIDINI` (account ID), `CARDNINI` (card number), `TTYPCDI`/`TCATCDI`/`TRNSRCI`/`TRNAMTI`/`TDESCI`/`TORIGDTI`/`TPROCDTI`/`MIDI`/`MNAMEI`/`MCITYI`/`MZIPI`, `CONFIRMI`
- `DFHCOMMAREA` ‚Äî CICS commarea with `CARDDEMO-COMMAREA`; `CDEMO-CT02-TRN-SELECTED` may pre-populate `CARDNINI`
- `CXACAIX` CICS VSAM AIX ‚Äî account‚Üícard cross-reference; used when account ID is provided (to resolve XREF-CARD-NUM)
- `CCXREF` CICS VSAM KSDS ‚Äî card‚Üíaccount cross-reference; used when card number is provided (to resolve XREF-ACCT-ID)
- `TRANSACT` CICS VSAM KSDS ‚Äî browsed backward to find max TRAN-ID; new record WRITEn here
- `ACCTDAT` CICS VSAM KSDS ‚Äî read to validate account existence

## Outputs
- New record in `TRANSACT` with all user-entered fields plus system-generated TRAN-ID
- `COTRN2AO` ‚Äî BMS map output with confirmation message or error
- `XCTL` to `COMEN01C` or `CDEMO-FROM-PROGRAM` (PF3)

## Key Business Rules
1. `EIBCALEN = 0` ‚Üí XCTL to COSGN00C (authentication required).
2. **Account/card resolution:** Either ACTIDINI or CARDNINI must be provided (not both, not neither). If account given ‚Üí reads CXACAIX to resolve card; if card given ‚Üí reads CCXREF to resolve account; both missing shows "Account or Card Number must be entered..." error.
3. All data fields are mandatory: type code, category code, source, description, amount, original date, processing date, merchant ID, merchant name, city, ZIP. Each blank shows a specific "can NOT be empty" error.
4. Amount must be numeric (validated via FUNCTION NUMVAL); dates validated via CALL to `CSUTLDTC` with `WS-DATE-FORMAT = 'YYYY-MM-DD'`; severity code `'0000'` = valid.
5. `CONFIRMI = 'Y'/'y'` ‚Üí `ADD-TRANSACTION` procedure; `'N'`/spaces ‚Üí "Confirm to add..." prompt; other ‚Üí "Invalid value" error.
6. New TRAN-ID generated by STARTBR/READPREV from HIGH-VALUES to find last TRAN-ID, then +1 (same pattern as COBIL00C).
7. PF4 clears the current screen. PF5 copies data from the last entered transaction into the form (useful for batch entry).
8. No update to ACCTDAT ‚Äî adding a transaction does not auto-update the account balance (contrast with COBIL00C which does).

## Notable COBOL Constructs
- **Bidirectional xref lookup:** Either CXACAIX (by account) or CCXREF (by card number) depending on which key was entered ‚Äî two separate CICS READ paths for the same logical resolution operation.
- **CSUTLDTC-PARM group:** Inline date validation parameter area (not via COPY); `CSUTLDTC-RESULT-SEV-CD PIC X(04)` checked for `'0000'`.
- **PF5 copy last transaction:** `COPY-LAST-TRAN-DATA` reads commarea's saved transaction ID and re-populates the screen from the last entered transaction ‚Äî no Java analogue; closest would be a "copy previous" button using session storage.

## Copybook Dependencies
- `COCOM01Y` ‚Äî `CARDDEMO-COMMAREA`; `CDEMO-CT02-INFO` appended inline
- `COTRN02` ‚Äî BMS mapset (COTRN2AI, COTRN2AO)
- `CVTRA05Y` ‚Äî `TRAN-RECORD`
- `CVACT01Y` ‚Äî `ACCOUNT-RECORD`
- `CVACT03Y` ‚Äî `CARD-XREF-RECORD` (XREF-ACCT-ID, XREF-CARD-NUM)
- `COTTL01Y`, `CSDAT01Y`, `CSMSG01Y` ‚Äî standard copybooks
- `DFHAID`, `DFHBMSCA` ‚Äî CICS constants

## Called Programs
- `CSUTLDTC` ‚Äî date validation (via CALL, not XCTL)

## Data Structures
| Field | PIC | Description |
|---|---|---|
| `WS-TRAN-ID-N` | `9(16)` | New transaction ID = last + 1 |
| `WS-TRAN-AMT-N` | `S9(9)V99` | Amount parsed via FUNCTION NUMVAL |
| `CSUTLDTC-RESULT-SEV-CD` | `X(04)` | `'0000'` = date valid |
| `WS-DATE-FORMAT` | `X(10) VALUE 'YYYY-MM-DD'` | Date format passed to CSUTLDTC |
